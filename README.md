# FrameWorks Release Manifests

Release manifests and version channels for the FrameWorks platform. Contains image digests, binary references, and channel pointers (stable/rc) — consumed by the CLI for deployments.

**This repo does not contain cluster configuration, server addresses, or infrastructure secrets.** Those live in a separate private repository.

## Structure

```
.
├── releases/          # Release manifests (one per version tag)
│   ├── v0.1.0-rc1.yaml
│   ├── v0.1.0-rc2.yaml
│   └── ...
└── channels/          # Channel pointers
    ├── stable.yaml    # Latest production-ready release
    └── rc.yaml        # Latest release candidate
```

## Release Manifests

Automatically generated by CI when a version tag is pushed to the monorepo. Each manifest contains:

- Platform version (vX.Y.Z)
- Git commit SHA + timestamp
- Service versions
- Docker image references with SHA256 digests
- Native binary download references
- External dependencies (MistServer, etc.)
- Tested infrastructure versions

## Channels

- **stable** — Latest production-ready release. Production clusters track this.
- **rc** — Release candidate. Staging clusters track this.

## Release Flow

1. **Tag**: Create annotated tag `vX.Y.Z` (GA) or `vX.Y.Z-rcN` (RC) in the monorepo.
2. **Build**: CI builds all services, pushes images, compiles native binaries.
3. **Manifest**: CI generates `releases/<tag>.yaml` with all artifact references and pushes here.
4. **Channel**: RC tags update `channels/rc.yaml`. GA tags update `channels/stable.yaml`.
5. **Deploy**: The CLI reads the manifest to pull images by digest and download binaries.

## Tag Types

| Tag pattern | Channel | Purpose |
|-------------|---------|---------|
| `vX.Y.Z-rcN` | rc | Release candidate for staging/testing |
| `vX.Y.Z` | stable | General availability for production |

Hotfixes: branch from the previous GA commit, fix, cut `vX.Y.(Z+1)` (optionally RC first for staging soak).
